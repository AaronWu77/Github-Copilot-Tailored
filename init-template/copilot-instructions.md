# Copilot Instructions Template

这个模板用于新仓库初始化时的 `copilot-instructions.md` 起始版本。

## Copilot-instructions.md的构建

1. 根据此文档进行仓库的初始化流程，创建 `.github/copilot-instructions.md`，其中内容结构在满足Github Copilot CLI的使用要求的前提下。尽可能的精简
2. 同时，在完成撰写之后，将本文档的 **Plan 模式约束**， **`/doc/plan.md` 固定结构**， **Autopilot 模式约束** 也放到仓库的 `.github/copilot-instructions.md` 中， 以符合使用需求

## Plan 模式约束

1. 进入 plan 模式后，任何方案都不能直接定稿。
2. 每当形成一个新的计划步骤、设计决策或执行方案时，必须先调用 `ask_user` 逐项询问用户是否接受该方案。
3. 只有在当前步骤获得用户明确确认后，才能进入下一个计划步骤。
4. 所有计划细节都确认完成后，除了更新会话内的计划文件外，还必须在仓库根目录创建 `/doc/plan.md`，并把最终计划写入该文件供用户审阅。
5. 如果已有 `/doc/plan.md`，则在每次计划更新后都要同步更新该文件内容，确保用户随时可以查看最新的计划细节。

## `/doc/plan.md` 固定结构

`/doc/plan.md` 必须按下面顺序组织内容：

1. **功能目的**：先说明这个计划要解决什么问题、目标是什么、范围边界是什么。
2. **TodoList**：列出需要执行的任务项，便于用户快速审阅整体工作拆分。
3. **具体执行方案**：说明每个阶段准备如何实施、涉及哪些模块、预期如何落地。

## Autopilot 模式约束

1. 进入 autopilot 模式后，不得直接开始执行。
2. 每次准备开始实施前，必须先调用 `ask_user`，明确询问用户是否按照当前仓库中的 `/doc/plan.md` 执行。
3. 只有在用户明确同意后，才能继续实现、修改代码、运行命令或推进任务。
4. 如果 `/doc/plan.md` 不存在、内容过期，或用户明确表示要调整方案，必须先回到计划确认流程，再继续执行。
5. 完成修改后提醒我运行 `/review`

## `/review` 指令约束

1. 每当用户要求 `/review` 时，检查目前 `/doc/plan.md` 以及 对话过程中发生的代码改动。
2. 如果审查代码无误，则在`.github/copilot-instructions.md`的最后按照已有版本号写上简要的更新日志，如果文档中尚未有更新日志则主动创建更新日志。更新日志格式如下：
```markdown
**v1**
[新增功能描述] 
[删减功能描述]
**v2**
[新增功能描述] 
[删减功能描述]
```
3. 如果审查代码有误，则调用 `ask_usr` 向用户反馈现有代码存在问题，并且在当前已有的 `/doc/plan.md` 新增一个代码审查部分，指出代码存在的问题，并且指出哪些todolist涵盖的部分出现问题。最后提醒用户调用 `/plan` 重新规划解决方案
